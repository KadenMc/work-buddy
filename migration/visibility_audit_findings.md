# Visibility audit findings

Generated by audit pass against six auto_run steps declared with `mode: "summary"` + all-keys `include_keys`, defeating the elision system.

---

```yaml
workflow_step: dev-document.scan
return_keys_with_sizes:
  changed_files:   "~1KB"
  classified:      "~1KB"
  subsystem_slugs: "~0.5KB"
  candidate_units: "~32KB"
  base_ref:        "~10B"
  warnings:        "~0.1KB"
recommended_mode: auto
include_keys_if_summary: []
justification: >
  candidate_units is the dominant cost (~32KB for a busy session) and varies
  with the number of matching knowledge units. auto picks full when the result
  is small and emits a manifest when large, covering both regimes correctly
  without hard-coding an include list.
```

```yaml
workflow_step: dev-document.validate
return_keys_with_sizes:
  total_units:  "~10B"
  checks_run:   "~0.2KB"
  passed:       "~10B"
  failed:       "~10B"
  summary:      "~0.3KB"
  errors:       "~0KB–50KB"
recommended_mode: summary
include_keys_if_summary: ["passed", "failed", "summary", "total_units"]
justification: >
  errors is the one variable-size field: zero items on a clean store, but
  potentially hundreds of dicts on a broken one (well over 10KB). The four
  included keys give the agent an immediate pass/fail verdict and a per-check
  breakdown; the full errors list is fetchable on demand and only needed when
  failed > 0.
```

```yaml
workflow_step: dev-commit.assess
return_keys_with_sizes:
  current_branch:  "~20B"
  is_main:         "~10B"
  changed_files:   "~0.5KB–5KB"
  classified:      "~0.5KB–3KB"
  test_candidates: "~0.1KB–1KB"
  warnings:        "~0.1KB–0.5KB"
recommended_mode: auto
include_keys_if_summary: []
justification: >
  No single key dominates unconditionally — all six are needed by the
  downstream reasoning step. In a small session the whole dict is well under
  10KB; in a large monorepo session with many changed files it may exceed the
  threshold. auto handles both: full when small, manifest when large.
```

```yaml
workflow_step: update-journal.read-journal
return_keys_with_sizes:
  target_date:     "~12B"
  ambiguous:       "~10B"
  hint:            "~0B–0.1KB"
  exists:          "~10B"
  created:         "~10B"
  error:           "~0B–0.2KB"
  collect_since:   "~25B"
  collect_until:   "~25B"
  last_log_ts:     "~25B"
  log_section:     "~0.1KB–20KB"
  sign_in_section: "~0.1KB–2KB"
recommended_mode: auto
include_keys_if_summary: []
justification: >
  log_section size is unbounded — a journal with a long running-notes carry-
  over can exceed 10KB. All keys are needed inline by the synthesis step, so
  there is no obviously droppable large key; auto correctly surfaces everything
  when the result is small and emits a manifest (prompting a targeted fetch)
  when it is large.
```

```yaml
workflow_step: stress-test.compute-primes
return_keys_with_sizes:
  prime_count:      "~10B"
  largest_prime:    "~10B"
  limit:            "~10B"
  elapsed_seconds:  "~10B"
recommended_mode: full
include_keys_if_summary: []
justification: >
  The function returns four scalar fields totalling well under 200 bytes.
  Note: the current workflows.json visibility block lists keys count, first,
  last, checksum — none of which exist in the actual return dict
  (prime_count, largest_prime, limit, elapsed_seconds). full is correct for
  this result shape and also eliminates the stale key mismatch.
```

```yaml
workflow_step: task-me.load-context
return_keys_with_sizes:
  status:               "~10B"
  current_contexts:     "~0.1KB"
  now_iso:              "~30B"
  errors:               "~0B–1KB"
  task_briefing:        "~2KB–30KB"
  calendar:             "~0.1KB–5KB"
  contract_constraints: "~0.5KB–5KB"
  engage:               "~2KB–40KB"
recommended_mode: auto
include_keys_if_summary: []
justification: >
  Both task_briefing and engage are variable-size payloads whose combined
  size scales with the number of tasks and contracts; on an active session
  the result easily exceeds 10KB. No single key is safely omittable (the
  engage step needs all of them), so auto is the right choice: full when
  the aggregate is small, manifest otherwise.
```
